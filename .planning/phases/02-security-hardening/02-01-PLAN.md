# Plan 02-01: Middleware & Route Protection

This plan establishes defense-in-depth routing protection, global account status verification, user authorization fallback handling, and endpoint rate limiting.

<threat_model>
| Threat | Classification | Mitigation |
| :--- | :--- | :--- |
| Spoofing | S | Authenticated users attempting to bypass role permissions are blocked by component-level gate checks on every interaction. |
| Tampering | T | Suspended or deactivated users are automatically signed out on the next page load using the CheckAccountStatus middleware. |
| Repudiation | R | Restricting mutation action access prevents unauthorized state-changing operations from occurring without valid session context. |
| Information Disclosure | I | Replacing standard HTTP 403 pages with a session-flashed toast notification redirects users to safe paths instead of displaying stack traces or layout structure. |
| Denial of Service | D | Enforcing specific rate limits on login, QR scanning, search, and transactions prevents bot scripts from overloading resources. |
| Elevation of Privilege | E | Enforcing permissions both at route-level middleware and at component-level execution protects against route manipulation attacks. |
</threat_model>

Create the CheckAccountStatus middleware. The handle() method must extract the authenticated user, check if their status is 'Active' (or similar column name, verified against database), and if not, call Auth::logout(), invalidate and regenerate the session, and redirect to the 'login' route with an error flash message.
In bootstrap/app.php, register CheckAccountStatus under the web middleware group. Configure custom rate limiters for 'login', 'qr-scanning', 'search', and 'transactions' inside withMiddleware() or routes/web.php as appropriate, and handle authorization exceptions (AccessDeniedHttpException/AuthorizationException) globally in withExceptions() to redirect unauthorized authenticated users to the dashboard route with a flashed mary.toast session payload, and guests to the login route. Add the throttle middleware limits to the appropriate routes in routes/web.php and routes/auth.php (e.g. login/auth routes, search, transactions, and qr endpoints).

Update all administrative Livewire components to perform gate checks in mount() and action methods.
In AdminComponent.php (base class), define an authorizeAccess() helper checking Gate::allows('librarian-or-admin-access'). If not authorized, throw an AuthorizationException (which will be caught by the global exceptions handler and redirected with a toast).
In specific sub-components, call authorizeAccess() in mount(). Ensure that components that override mount() (specifically AdminRuleAndRegulationIndex, CreateAcademicPaper, and EditAcademicPaper) explicitly call the base authorization check or authorizeAccess() inside their mount() override.
For sensitive actions (e.g. updating a role in AdminManageRoles, creating/editing academic papers, confirming borrows, log deletion), perform specific Gate checks (e.g., Gate::authorize('manage-user-roles'), Gate::authorize('manage-academic-papers')) at the beginning of the action method before any database mutation.
