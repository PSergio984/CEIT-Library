# Coding Conventions

**Analysis Date:** 2025-05-14

## Naming Patterns

**Files:**
- Classes: PascalCase (e.g., `app/Models/User.php`, `app/Livewire/QrScanner.php`)
- Tests: PascalCase with `Test` suffix (e.g., `tests/Feature/AcademicPaperFormTest.php`)
- Views: kebab-case (e.g., `resources/views/livewire/pages/admin/dashboard.blade.php`)
- Config: snake_case (e.g., `config/departments.php`)

**Functions:**
- Class Methods: camelCase (e.g., `handleScan`, `processAttendance`)
- Test Methods: snake_case (e.g., `admin_can_create_a_new_academic_paper`) or prefixed with `test_`
- Helper Functions: snake_case

**Variables:**
- Local Variables: camelCase (e.g., `decryptedData`)
- Public Component Properties: camelCase (e.g., `isScanning`, `hasError`)
- Private/Protected Properties: camelCase

**Types:**
- Classes/Interfaces: PascalCase
- Enums: PascalCase (e.g., `InventoryStatus`)
- Scalar Types: PHP 8.2 type hints are used throughout (`string`, `bool`, `int`, `?string`, `void`)

## Code Style

**Formatting:**
- Tool: `laravel/pint`
- Rules: Default Laravel preset. Run using `vendor/bin/pint --dirty`.
- Strict use of curly braces for control structures.

**Linting:**
- Handled by Laravel Pint.
- PHP 8.2 features like constructor property promotion and explicit return types are required.

## Import Organization

**Order:**
1. App-specific namespaces (`App\Models`, `App\Traits`, `App\Http`)
2. Standard libraries and Third-party packages (`Carbon\Carbon`, `Livewire\Component`)
3. Laravel Facades (`Illuminate\Support\Facades\Log`)
4. Internal namespace imports are typically alphabetical within their groups.

**Path Aliases:**
- Standard PSR-4 mapping: `App\` -> `app/`, `Database\Factories\` -> `database/factories/`, `Database\Seeders\` -> `database/seeders/`, `Tests\` -> `tests/`.

## Error Handling

**Patterns:**
- `try-catch` blocks for operations involving external data or complex logic (e.g., QR decryption, attendance processing).
- Use of `Log::error()` or `Log::warning()` to record exceptions with context.
- User-facing errors are reported via Livewire/Mary UI Toast notifications (`$this->error($message, $title)`).

## Logging

**Framework:** Laravel standard logging (`Log` facade).

**Patterns:**
- Log info for successful critical operations (e.g., "Attendance recorded successfully").
- Log warnings for expected but failed operations (e.g., "Attendance processing failed").
- Log errors for unexpected exceptions with full stack trace or exception object in context.

## Comments

**When to Comment:**
- Complexity: Use comments to explain non-obvious logic or legacy field compatibility.
- Documentation: Use PHPDoc for class properties and methods (often generated by `ide-helper`).

**JSDoc/TSDoc:**
- Not strictly enforced for JS as most logic resides in PHP (Livewire/Volt), but standard comments used in Alpine.js scripts if present.

## Function Design

**Size:** Prefer small, single-responsibility methods. Large operations (like QR processing) are broken down into private helper methods (`decryptAndValidateAttendanceData`, `processAttendance`).

**Parameters:** Explicit type hints required.

**Return Values:** Explicit return type declarations required.

## Module Design

**Exports:** Standard PSR-4 class structure.

**Barrel Files:** Not used.

**Traits:** Used heavily for shared logic (e.g., `Toast`, `CreatesQrCanonicalMessage`, `CreatesTestDatabase`).
